# 要件・設計書

## 1. 要件定義

### 1.1 基本情報
- ソフトウェア名称: hatakei（はたけい）
- リポジトリ名: hatakei

### 1.2 プロジェクト概要

家庭菜園のスケジュールと畝のレイアウトを計画するためのWebアプリ。
タイムライン（栽培スケジュール管理）と畝マップ（レイアウト管理）の2つのビューを組み合わせ、
タイムラインのバーが乗っている時点の畝の状態をマップ上にリアルタイム反映する。

### 1.3 機能要件

#### 1.3.1 マップ機能
- 60×60グリッドのフィールド（1マス=30cm×30cm、全体18m×18m）
- 畝ブロック（1マス幅×nマス長、可変）のグリッド上への配置・移動・削除
- 畝の向き: 縦・横どちらも配置可能、90度回転対応
- 自由配置（通路は任意グリッドで表現）
- タイムラインのバー位置に対応した野菜アイコンの表示
- 畝への名前付け

#### 1.3.2 タイムライン機能
- 縦軸: 時間軸（週単位、3月第1週〜2月最終週、52週）
- 横軸: 畝（1畝=1列、横スクロール対応）
- マップへの畝追加と連動して列が追加される（1対1対応）
- 栽培ブロックの配置:
  - プリセットパネルからドラッグ＆ドロップ
  - テキスト入力による日付・期間指定（Googleカレンダー方式）
- 栽培ブロックの移動・リサイズ・削除
- バー（現在表示時点）:
  - 手動ドラッグで移動
  - 「今日」ボタンで現在日付にジャンプ

#### 1.3.3 野菜プリセット機能
- 初期プリセット: サーバーDB管理（温暖地/中部地方基準）
  - 野菜名、アイコン、種まき時期（週範囲）、収穫時期（週範囲）、栽培期間、カテゴリ
- ユーザーカスタムプリセット: ローカルストレージ管理
  - 初期プリセットと同形式で作成・編集・削除可能

#### 1.3.4 データ永続化
- マップレイアウト（畝配置）→ ローカルストレージ
- タイムライン配置（栽培ブロック）→ ローカルストレージ
- ユーザーカスタムプリセット → ローカルストレージ
- 野菜プリセット（初期データ）→ サーバーDB（読み取り専用API）

### 1.4 非機能要件

#### 1.4.1 性能要件
- タイムラインのバードラッグ時、マップへの反映が60fps以上でスムーズに動作すること
- 初期ロード時のプリセットデータ取得は1秒以内

#### 1.4.2 対応環境
- PC・モバイル両対応のレスポンシブレイアウト
- レイアウト: 上1/3にマップ、下2/3にタイムライン（PC・モバイル共通）
- モダンブラウザ（Chrome/Firefox/Safari/Edge 最新版）

#### 1.4.3 運用・保守要件
- ローカルストレージのデータ形式にバージョン管理を設け、将来の構造変更に対応できること
- APIエラー時はフォールバックとしてバンドル済み静的プリセットを表示すること

### 1.5 制約条件
- ユーザー認証なし（シングルユーザー前提）
- データのサーバー保存なし（クラウド同期なし）
- 混植なし（1畝1時期1野菜）
- 温暖地（中部地方）の栽培カレンダーを基準とする

### 1.6 開発環境
- 言語: TypeScript
- フロントエンド: Vite + React
- D&Dライブラリ: @dnd-kit
- 状態管理: Zustand
- スタイル: Tailwind CSS
- バックエンド: Hono
- DB: PostgreSQL
- Hosting（フロントエンド）: Vercel
- Hosting（バックエンド）: Railway

### 1.7 成果物
- ソースコード（フロントエンド・バックエンド）
- 設計書（本ドキュメント）
- テストコード
- README（セットアップ・デプロイ手順含む）

## 2. システム設計

### 2.1 システム概要設計

#### 2.1.1 システムアーキテクチャ

```
[Browser]
  ├─ マップコンポーネント     ←→ [Zustandストア] ←→ [localStorage]
  ├─ タイムラインコンポーネント ←→ [Zustandストア] ←→ [localStorage]
  └─ プリセットパネル         ←→ [Zustandストア] ←→ [localStorage]（カスタム）
                                       ↕ REST API (fetch)
                             [Hono APIサーバー on Railway]
                                       ↕
                             [PostgreSQL on Railway]（プリセットデータ）
```

#### 2.1.2 主要コンポーネント

1. **マップグリッド**
   - 60×60グリッドの描画
   - 畝ブロックの配置・移動（@dnd-kit）
   - バー時点の野菜アイコン表示

2. **畝ブロック**
   - 畝ブロックの表示（1×n グリッド、縦横対応）
   - 野菜アイコンのオーバーレイ

3. **タイムライン**
   - 52週 × N列の描画（横スクロール）
   - 栽培ブロックの配置・移動・リサイズ（@dnd-kit）
   - タイムラインバーの描画・ドラッグ

4. **栽培ブロック**
   - 栽培期間の矩形表示
   - 野菜名・アイコンの表示

5. **プリセットパネル**
   - 野菜プリセット一覧（初期 + カスタム）
   - ドラッグソース

6. **タイムラインバー**
   - 横断バーの描画
   - 現在週の表示
   - 「今日」ボタン

#### 2.2.1 型定義

// 野菜プリセット

### 2.3 インターフェース設計

#### 2.3.1 レイアウト

```
┌─────────────────────────────────┐
│  マップグリッド (上 1/3)          │
│  [畝ブロック + 野菜アイコン]        │
├─────────────────────────────────┤
│  タイムライン (下 2/3)            │
│  [バー + 栽培ブロック + プリセット] │
└─────────────────────────────────┘
```

#### 2.3.2 マップ操作
- 畝の追加: 空きグリッドをクリック → 長さ・名前を入力して確定
- 畝の移動: D&Dでグリッド上を移動
- 畝の回転: 畝を右クリック → 回転
- 畝の削除: 畝を右クリック → 削除

#### 2.3.3 タイムライン操作
- 栽培ブロックの追加: プリセットパネルから列へD&D / 列をクリックしてテキスト入力
- 栽培ブロックの移動: D&Dで列内・列間を移動
- 栽培ブロックのリサイズ: ブロック端をドラッグ
- バーの移動: D&Dで週をまたいで移動
- 今日へジャンプ: 「今日」ボタンをクリック

### 2.4 セキュリティ設計
- バックエンドはCORS設定でVercelのドメインのみ許可
- 入力値（野菜名等）はXSS対策としてReactのエスケープに依存（dangerouslySetInnerHTMLは使用しない）

### 2.5 テスト設計
- ユニットテスト（Vitest）

- コンポーネントテスト（Vitest + Testing Library）
  - マップグリッド: 畝の配置・移動
  - タイムライン: ブロックの配置・バー操作
- APIテスト

### 2.6 開発工程

#### 2.6.1 開発フェーズ

1. **Phase 1: 基盤構築**
   - モノレポ構成の初期化（frontend / backend）
   - DB・Railway・Vercel環境のセットアップ
   - 型定義・Zustandストアの実装
   - ローカルストレージ永続化の実装

2. **Phase 2: マップ機能**
   - 60×60グリッドの描画
   - 畝ブロックの配置・移動・回転（@dnd-kit）

3. **Phase 3: タイムライン機能**
   - 52週 × N列の描画
   - 栽培ブロックの配置・移動・リサイズ（@dnd-kit）
   - バーのドラッグ・「今日」ボタン

4. **Phase 4: プリセット機能**
   - バックエンドAPI実装（Hono + PostgreSQL）
   - プリセットデータのシード投入（主要野菜30種程度）
   - プリセットパネル・カスタム追加UI

5. **Phase 5: 連携・仕上げ**
   - バー移動とマップの野菜アイコン連動
   - レスポンシブ対応（PC・モバイル）
   - エラーハンドリング・フォールバック

#### 2.6.2 マイルストーン
- M1: マップ上で畝を自由配置できる
- M2: タイムラインで栽培ブロックをD&D配置できる
- M3: バーを動かすとマップの野菜アイコンが更新される
- M4: プリセットからD&Dでタイムラインに配置できる
- M5: モバイルで一通りの操作が完結する

#### 2.6.3 リスク管理
- @dnd-kitのモバイルタッチ対応の制約
  - 対応策: 早期にモバイル動作を検証し、必要なら操作UXを調整
- ローカルストレージの容量制限（約5MB）
  - 対応策: データ構造を最小化、将来的にIndexedDBへ移行検討
- 月の週数の揺れ（4週/5週）
  - 対応策: 週インデックス仕様を固定し、月またぎはインデックスで管理
